# Phase 10.1: Board Evolution — Task Board Polish + Process Board — Research

**Researched:** 2026-03-05
**Domain:** Frontend Kanban UI (Vue 3 + PrimeVue 4 + Chart.js 4), Backend Board domain extension (Symfony 8 + Doctrine)
**Confidence:** HIGH (all findings verified against actual codebase; no external library research required — all dependencies already present)

---

## Summary

Phase 10.1 has two distinct halves. The first — Task Board Polish — is a pure frontend enhancement of the existing KanbanBoardPage.vue. The existing backend and domain are complete; the board already stores wipLimit, statusMapping, and column color. What is missing is: (a) richer kanban cards (avatar badge, comment count, label dots), (b) swimlane grouping logic (by assignee or priority), (c) a URL-persisted Quick Filter bar, and (d) proper WIP limit color semantics (amber at 80%, red at 100%). All these require only frontend changes and two small backend additions (comment count on TaskDTO, assignee avatar URL on TaskDTO).

The second half — Process Board — is a new board type that is architecturally distinct from the existing Task Board. A Process Board uses a published ProcessDefinition as its "template": its columns are auto-generated from the definition's `task` nodes in topological order (Start → tasks → Completed). Cards are live ProcessInstance rows, not Task rows. Dragging a card triggers `ExecuteTaskAction` via the existing workflow engine. This requires: (a) a new `board_type` discriminator on the `Board` entity (task_board vs process_board), (b) a new column type for process boards that stores a `node_id` instead of `status_mapping`, (c) a new backend endpoint that generates process board columns from a definition's nodesSnapshot, (d) a new `ProcessBoardCard` frontend component, and (e) a pipeline metrics computation using the existing audit_log table.

The primary risk is the topological sort for column ordering (Start → tasks → Completed). The existing `ProcessGraph` class provides `outgoingTransitions()` and `incomingTransitions()` — a simple BFS/Kahn's algorithm on top of these is straightforward. No external library is needed.

**Primary recommendation:** Extend the existing Board entity with a `board_type` discriminator column, reuse `ProcessGraph::fromSnapshot()` for topological ordering, and treat `ExecuteTaskAction` as the drag-complete handler. Do not invent a new task execution path.

### Pattern 1: Swimlane Grouping (Pure Frontend Computed)

**What:** Group filtered tasks into swimlane rows using a `computed` that returns `Array<{ key: string, label: string, tasks: TaskDTO[] }>`.
**When to use:** BRD-02

```typescript
// In KanbanBoardPage.vue
const swimlaneMode = ref<'none' | 'assignee' | 'priority'>('none')

const swimlanes = computed(() => {
  if (swimlaneMode.value === 'none') return [{ key: 'all', label: '', tasks: filteredTasks.value }]

  if (swimlaneMode.value === 'assignee') {
    const groups = new Map<string, { label: string; tasks: TaskDTO[] }>()
    for (const task of filteredTasks.value) {
      const key = task.assigneeId ?? 'unassigned'
      const label = task.assigneeName ?? t('kanban.unassigned')
      if (!groups.has(key)) groups.set(key, { label, tasks: [] })
      groups.get(key)!.tasks.push(task)
    }
    // Ensure "Unassigned" row always last
    const sorted = [...groups.entries()].sort(([a], [b]) => {
      if (a === 'unassigned') return 1
      if (b === 'unassigned') return -1
      return 0
    })
    return sorted.map(([key, v]) => ({ key, label: v.label, tasks: v.tasks }))
  }

  // priority mode: Critical → High → Medium → Low
  const PRIORITY_ORDER = ['critical', 'high', 'medium', 'low']
  return PRIORITY_ORDER.map((p) => ({
    key: p,
    label: t(`tasks.priority_${p}`),
    tasks: filteredTasks.value.filter((t) => t.priority === p),
  }))
})
```

### Pattern 2: URL Query Param Persistence for Filters

**What:** Sync filter state to URL via `useRoute`/`useRouter` `watch`. This is the same pattern used in `ProcessInstancesPage.vue` (Phase 6 process-instance pagination).
**When to use:** BRD-03

```typescript

const filterAssignee = ref(route.query.assignee as string ?? '')
const filterLabels = ref<string[]>((route.query.labels as string)?.split(',').filter(Boolean) ?? [])

const filterDueDateTo = ref(route.query.to as string ?? '')

  router.replace({
    query: {
      ...route.query,
      q: filterText.value || undefined,
      assignee: filterAssignee.value || undefined,
      labels: filterLabels.value.length ? filterLabels.value.join(',') : undefined,
      from: filterDueDateFrom.value || undefined,
      to: filterDueDateTo.value || undefined,
    },
  })
}, { deep: true })
```

### Pattern 4: Topological Sort for Process Board Columns (BRD-05)

**What:** Given the nodesSnapshot from ProcessDefinitionVersion, sort `task` nodes in execution order using Kahn's algorithm on the ProcessGraph.
**When to use:** BRD-05 — backend endpoint that generates Process Board columns

```php
// In CreateProcessBoardHandler or a domain service
private function topologicalTaskOrder(array $snapshot): array
{
    $graph = ProcessGraph::fromSnapshot($snapshot);
    $nodes = $snapshot['nodes'];
    $transitions = $snapshot['transitions'];

    // Build in-degree map
    $inDegree = [];
    $adjList = [];
    foreach ($nodes as $node) {
        $inDegree[$node['id']] = 0;
        $adjList[$node['id']] = [];
    }
    foreach ($transitions as $t) {
        $inDegree[$t['target_node_id']]++;
        $adjList[$t['source_node_id']][] = $t['target_node_id'];
    }

    // Kahn's BFS
    $queue = [];
    foreach ($inDegree as $nodeId => $deg) {
        if (0 === $deg) $queue[] = $nodeId;
    }

    $ordered = [];
    while ([] !== $queue) {
        $nodeId = array_shift($queue);
        $ordered[] = $nodeId;
        foreach ($adjList[$nodeId] as $neighbor) {
            $inDegree[$neighbor]--;
            if (0 === $inDegree[$neighbor]) {
                $queue[] = $neighbor;
            }
        }
    }

    // Return only 'task' nodes in topological order
    $nodeMap = array_column($nodes, null, 'id');
    return array_values(array_filter(
        array_map(static fn ($id) => $nodeMap[$id] ?? null, $ordered),
        static fn ($n) => null !== $n && 'task' === $n['type'],
    ));
}
```

### Pattern 7: Process Board Data Query (BRD-06)

**What:** Fetch ProcessInstances for a given definition, enriched with active task assignee name. Use DBAL (not ORM) since this joins `workflow_process_instances_view` + `task_manager_tasks`.

```php
// In GetProcessBoardDataHandler

    ->setParameter('defId', $query->processDefinitionId)
    ->executeQuery()
    ->fetchAllAssociative();
```

Note: The existing `workflow_process_instances_view` stores `tokens` as JSONB. The active node can be read by finding the first token with `status = 'waiting'` or `status = 'active'` in that JSONB.

### Pattern 8: Stage Timing Metrics from Audit Log (BRD-08)

**What:** Compute average time per stage using `process.started` and `process.completed` events from `audit_log`. For stage-level timing, use `task.status_changed` events filtered by `changes->>'to' = 'done'` with organization scope.

The audit_log stores:
- `event_type = 'process.started'`, `entity_type = 'process_instance'`, `occurred_at`
- `event_type = 'process.completed'`, `entity_type = 'process_instance'`, `occurred_at`

**Stage-level timing gap:** There is no `task.node_activated` or per-node timing event in audit_log. The `OnTaskNodeActivated` event is a domain event handled by the workflow engine but is NOT audited. Only `process.started` and `process.completed` give timestamps. Per-stage average requires an `OnTaskStatusChangedAudit` cross-reference with workflow_task_links to get the node_id. This is complex.

**Practical approach for BRD-08:** The "average time per stage" can be approximated by:
1. Overall process cycle time = `process.completed.occurred_at - process.started.occurred_at`
2. Tasks per stage are already tracked via `task.status_changed` (changes JSONB has `from`/`to`)

However, since no node_id is stored in audit_log for tasks, true per-BPMN-node timing requires either a new `task.node_activated` audit event or a JOIN through `workflow_task_links → tokens → nodesSnapshot`.

**Recommendation:** For the initial implementation of BRD-08, expose process-level metrics only (total active, completed per day = sparkline) and document per-stage averaging as a known gap. This avoids schema changes to audit_log.

### Pattern 9: ActionFormDialog Inline for Process Board Drag (BRD-07)

**What:** Reuse the existing `ActionFormDialog.vue` component. When drag is dropped on next stage column:
1. Call `GetTaskWorkflowContext` to get the active task's form schema for the default action
2. If form has required fields → show ActionFormDialog inline
3. If no required fields → call `ExecuteTaskAction` immediately with `{}`
4. If no valid transition → toast warning + reject drag

```typescript
// In ProcessBoardPage.vue
async function onProcessCardDrop(instanceId: string, targetColumn: ProcessBoardColumnDTO) {
  const instance = instances.value.find(i => i.id === instanceId)
  if (!instance || !instance.active_task_id) {
    toast.add({ severity: 'warn', detail: t('processBoard.noActiveTask'), life: 3000 })
    return
  }

  // Check if task has required form fields for default action
  const context = await taskApi.getWorkflowContext(orgId.value, instance.active_task_id)
  const defaultAction = context?.form_schema?.actions[0]
  const hasRequired = defaultAction?.form_fields.some(f => f.required)

  if (hasRequired) {
    selectedTaskForAction.value = instance.active_task_id
    selectedActionKey.value = defaultAction!.key
    showActionDialog.value = true
  } else {
    await executeTaskAction(instance.active_task_id, defaultAction!.key, {})
  }
}
```

### Anti-Patterns to Avoid

- **Loading all tasks for a Process Board the same way as Task Board:** Process boards fetch ProcessInstance rows, not Task rows. Do not reuse `taskStore.fetchTasks()` for Process Board.
- **Storing swimlane mode in Pinia store:** Swimlane mode is board-specific UI state, not global. Store in component `ref` + URL param.
- **Reading nodesSnapshot from live Node entities for topological sort:** Always use `ProcessDefinitionVersion.nodesSnapshot()` — same source used by the engine. Querying `Node` entity directly may miss version pinning.
- **Using `statusMapping` column for Process Board columns:** Process boards use `nodeId`, not `statusMapping`. The BoardColumn entity needs both fields, and the frontend must differentiate.
- **Calling `taskStore.transitionTask()` from Process Board drag:** Process boards advance via `ExecuteTaskAction` (workflow engine path), not `TransitionTask` (direct task state machine). Wrong path = token not moved.

## Common Pitfalls

### Pitfall 1: Comment Count Missing from TaskDTO
**What goes wrong:** BRD-01 requires comment count on kanban cards. `TaskDTO` has no `commentCount` field. If added naively with `SELECT COUNT(*) FROM task_manager_comments WHERE task_id = ?` per task, it causes N+1.
**Why it happens:** The original TaskDTO was designed before comment count on cards was needed.
**How to avoid:** Batch-load via a single DBAL query: `SELECT task_id, COUNT(*) FROM task_manager_comments WHERE task_id IN (...ids) GROUP BY task_id`. Add to `ListTasksHandler` alongside the existing label batch-load pattern.
**Warning signs:** If ListTasksHandler calls `$commentRepository->countByTaskId()` inside the `array_map` loop — that's N+1.

### Pitfall 2: Assignee Avatar URL Missing from TaskDTO
**What goes wrong:** BRD-01 requires avatar badge on kanban cards. `TaskDTO.assigneeName` exists but `assigneeAvatarUrl` does not. The `UserDTO` has `avatarUrl` (Phase 7), but it is not propagated to `TaskDTO`.
**Why it happens:** Phase 7 note: "TaskDetailSidebar shows current user's avatar only when task.assigneeId === auth.user.id — avoids premature DTO extension before Phase 8".
**How to avoid:** Extend `OrganizationQueryPort.resolveEmployeeDisplayNames()` to return `array<string, array{name: string, avatarUrl: ?string}>` and extend `DoctrineOrganizationQueryAdapter` to join Identity's `UserRepository` for avatarUrl. Add `?string $assigneeAvatarUrl = null` to `TaskDTO`.
**Warning signs:** avatarUrl is fetched per-card on frontend — that's a separate per-card API call.

### Pitfall 3: Process Board Drag Using Wrong Execution Path
**What goes wrong:** Developer uses `taskStore.transitionTask(orgId, taskId, 'workflow_complete')` from drag on Process Board. This calls the direct task state machine transition — it does NOT advance the workflow token or move the process to the next stage.
**Why it happens:** The task board drag uses `transitionTask()` for column changes. Developer assumes same pattern applies to process board.
**How to avoid:** Process Board drag must call `ExecuteTaskAction` (POST `/organizations/{orgId}/tasks/{taskId}/workflow-action`) with the action key. See `ExecuteTaskActionHandler` — it moves the token AND completes the task via `TransitionTaskCommand`.
**Warning signs:** Token stays at the same node after drag. ProcessInstance remains on same stage.

### Pitfall 4: Active Node Detection from `tokens` JSONB
**What goes wrong:** ProcessInstanceDTO contains `tokens` as `array<ProcessInstanceTokenDTO>` where each token has `node_id` and `status`. To find the active node, filter for `status = 'waiting'`. However, a process can have multiple waiting tokens (parallel branches). In that case, "current stage" is ambiguous.
**Why it happens:** Parallel branches are possible even though XOR gateways are the primary routing.
**How to avoid:** For Process Board cards (BRD-06), display the first waiting token's node name. For drag (BRD-07), if multiple waiting tokens exist, disable drag and show tooltip "Multiple active stages". For the initial implementation, document this limitation.
**Warning signs:** ProcessInstance appears twice in different columns.

### Pitfall 5: Topological Sort on Cyclic Graphs
**What goes wrong:** Designer allows creating transitions that could create cycles (e.g., "reopen" going back to a previous stage). Kahn's algorithm returns incomplete results for cyclic graphs.
**Why it happens:** BPMN allows loop-back transitions for rework patterns.
**How to avoid:** After Kahn's BFS, check `count($ordered) < count($nodes)`. If true, there is a cycle — fall back to position-based ordering (node.position_x ascending). The `ProcessGraphValidator` in the backend already validates acyclicity for execution, but the board column generator should be defensive.
**Warning signs:** Some task nodes are missing from the generated columns.

### Pitfall 6: WIP Limit Amber Color Variable
**What goes wrong:** `--p-amber-500` CSS variable may not exist in PrimeVue Aura theme (only `--p-yellow-500` or `--p-orange-500` might be present).
**Why it happens:** PrimeVue Aura theme uses a specific color palette. Not all Tailwind-style names exist.
**How to avoid:** Use `--p-orange-500` (already referenced in ChartsWidget.vue color map) for "warning" state. Check existing CSS variables in ChartsWidget.vue: `orange400: get('--p-orange-400')` confirms this variable exists.
**Warning signs:** Column header shows no color change despite warning state.

### Pitfall 7: Process Board Metrics — Per-Stage avg from audit_log
**What goes wrong:** BRD-08 asks for "average time per stage". The audit_log does not store `node_id` for task events — only `entity_type = 'task'` with `changes = {from, to}`. The workflow task link stores `node_id` + `process_instance_id` but is in a separate table.
**Why it happens:** Audit log was designed for actor attribution and change tracking, not performance profiling.
**How to avoid:** For BRD-08, scope metrics to process-level only: total active instances (from `workflow_process_instances_view WHERE status = 'running'`), throughput sparkline (from `audit_log WHERE event_type = 'process.completed'` in last 14 days grouped by day). Document "per-stage timing" as Phase 10.2 stretch goal requiring `task.node_activated` audit events.
**Warning signs:** Trying to JOIN `audit_log` + `workflow_task_links` + `nodesSnapshot` JSONB — this is complex and fragile.

## Open Questions

1. **Multiple active tokens on Process Board card**
   - What we know: ProcessInstance.tokens is JSONB array; multiple tokens can be waiting simultaneously (XOR gateway won't split but sub-processes can)
   - What's unclear: How to represent a ProcessInstance with 2 waiting tokens on the board (which column?)
   - Recommendation: For initial implementation, pick the first waiting token's node. Show a badge "2 active stages" on the card. Document as known limitation.

2. **Process Board — is board_type on the Board entity enough, or do we need a separate ProcessBoard entity?**
   - What we know: The Board entity is simple (id, orgId, name, description). Extending with `board_type` + `processDefinitionId` adds 2 nullable fields. The `BoardColumn` entity needs `nodeId` nullable alongside `statusMapping`.
   - What's unclear: Whether the planner wants a full domain redesign (new entity hierarchy) or pragmatic extension
   - Recommendation: Pragmatic extension (discriminator column + nullable fields). Full entity hierarchy (Board → TaskBoard, ProcessBoard) is overengineering for 2 board types.

3. **Should process board data come from a new endpoint or extend the existing board API?**
   - What we know: Existing `GET /organizations/{orgId}/boards/{boardId}` returns board+columns but no process instances
   - What's unclear: Whether to add a separate `GET /boards/{boardId}/instances` endpoint or embed in the board response
   - Recommendation: Separate endpoint `GET /organizations/{orgId}/boards/{boardId}/process-data` returning `{ instances: [...], metrics: {...} }`. Board metadata stays separate.

4. **Drag target validation — how to determine "next valid stage"?**
   - What we know: Process Board columns correspond to node_ids. Dragging card from column A to column B should trigger a workflow action. The workflow engine validates via `executeAction()`.
   - What's unclear: Whether to validate at frontend (check if transition exists from current node to target node) or backend (let engine reject with 422)
   - Recommendation: Backend validation only — let engine throw `WorkflowExecutionException`. Frontend catches 422/400 and shows toast. Don't duplicate transition validation in frontend.
