    **3. Create GetProcessBoardDataHandler** — inject `Connection` (DBAL), `BoardRepositoryInterface`.
    Steps:
    a. Load board to get `processDefinitionId`. Throw 400 if board is not process_board type.
    b. **Fetch instances:** DBAL query on `workflow_process_instances_view`:
       ```sql
       SELECT pi.id, pi.status, pi.started_at, pi.definition_name, pi.tokens
       FROM workflow_process_instances_view pi
       WHERE pi.organization_id = :orgId AND pi.definition_id = :defId
       ```
    c. For each instance, parse `tokens` JSONB to find the first token with `status = 'waiting'` — extract its `node_id` as `activeNodeId`.
    d. Resolve `activeNodeName` by loading the board's process definition version `nodesSnapshot` and looking up the node name by ID. Cache the snapshot since it is the same for all instances on this board.
    e. **Fetch active tasks:** DBAL query joining `workflow_task_links` + `task_manager_tasks` to get the task assigned to the active node for each instance:
       ```sql
       SELECT wtl.process_instance_id, t.id as task_id, t.assignee_id
       FROM workflow_task_links wtl
       JOIN task_manager_tasks t ON t.id = wtl.task_id
       WHERE wtl.process_instance_id IN (?) AND t.status NOT IN ('done', 'cancelled')
       ```
       Use ArrayParameterType::STRING for the IN clause.
    f. Batch-resolve assignee display names using `OrganizationQueryPort.resolveEmployeeDisplayNames()` — use the `['name']` key from the updated return type (Plan 01 dependency). **NOTE:** If Plan 01 has not yet been executed, use the existing string return type and adapt. The handler should handle both cases gracefully, or if the port returns strings, wrap them.
       **Actually:** Plan 03 is Wave 1 (parallel with Plan 01). So the OrganizationQueryPort may NOT yet be updated. Use a simpler approach here: inject `Connection` and do a direct DBAL query to resolve assignee names from `organization_employees` JOIN `identity_users` for the batch of assignee IDs. This avoids coupling to Plan 01's port change.
    g. **Fetch metrics:**
       - `totalActive`: COUNT from `workflow_process_instances_view WHERE status = 'running' AND definition_id = :defId AND organization_id = :orgId`.
       - `completedByDay`: from `audit_log` table:
         ```sql
         SELECT DATE(occurred_at) as d, COUNT(*) as cnt
         FROM audit_log
         WHERE event_type = 'process.completed'
           AND organization_id = :orgId
           AND changes->>'definition_id' = :defId
           AND occurred_at >= :since
         GROUP BY DATE(occurred_at)
         ORDER BY d
         ```
         Where `:since` = 14 days ago. If `definition_id` is not in audit_log changes, filter by `entity_type = 'process_instance'` and join with `workflow_process_instances_view` to check definition_id. **Simpler approach:** just query all `process.completed` events for the org in last 14 days and count — process board can show org-wide throughput since per-definition filtering in audit_log is complex (no definition_id in changes JSONB consistently). Document this as a known simplification.

      public function createProcessBoard(string $organizationId, Request $request): JsonResponse
