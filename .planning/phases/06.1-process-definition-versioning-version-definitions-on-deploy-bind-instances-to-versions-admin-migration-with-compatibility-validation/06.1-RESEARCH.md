# Phase 06.1: Process Definition Versioning — Research

**Researched:** 2026-03-01
**Domain:** BPM process versioning — admin migration UI, compatibility validation, event-sourced instance binding
**Confidence:** HIGH — findings based entirely on direct codebase inspection

---

## Summary

Phase 06.1 was inserted as "URGENT" after Phase 6 completed. Contrary to what the phase title implies, the **backend infrastructure for versioning is already fully implemented**. Phase 6 (plans 01 and 02) built: `ProcessDefinitionVersion` entity, `workflow_process_definition_versions` table, version snapshots on publish, `ProcessInstance.versionId` binding, `ListVersionsHandler`, `MigrateProcessInstancesHandler`, and the migration API endpoint. The entire engine already reads from the version snapshot for token execution, graph queries, task form schema, and task creation.

What is **actually missing** is:

1. A **frontend admin UI** to view version history and trigger migration — the API exists but no Vue page/component exposes it
2. The `MigrateProcessInstances` command currently only updates the **read model** (`workflow_process_instances_view`). When the engine reconstitutes a `ProcessInstance` from the event store, `versionId` comes from the immutable `ProcessStartedEvent` — not from the read model. This means migrated instances resume execution against the **old version graph** after migration. This is a correctness gap.
3. The compatibility validation in `MigrateProcessInstancesHandler` only checks that active token node IDs exist in the target version. It does NOT validate form field schema compatibility (e.g., a required field added to a transition that wasn't there before).

**Primary recommendation:** Decide the migration semantics first (read-model-only vs. event-sourced migration event), then build the admin frontend. Do NOT change the event store if event immutability is a design invariant — instead introduce a `ProcessInstanceMigratedEvent` that the engine applies to update `versionId` in the aggregate.

## The Real Gaps — What Phase 06.1 Must Build

### Gap 1: Migration Semantics — Event Store vs. Read Model Only (HIGH PRIORITY)

**Current behavior:** `MigrateProcessInstancesHandler` executes:
```sql
UPDATE workflow_process_instances_view SET version_id = :targetVersionId
WHERE definition_id = :definitionId AND status = 'running'
```

The event store (`workflow_process_events`) is immutable. When the engine loads an instance via `EventSourcedProcessInstanceRepository::findById()`, it calls `ProcessInstance::reconstitute(events)` which replays `ProcessStartedEvent` — and `applyProcessStarted()` sets `$this->versionId = ProcessDefinitionVersionId::fromString($event->versionId)`. This is the **original version**, not the migrated one.

**Impact:** After migration, when a task completes and `CompleteTaskNodeHandler` runs:
```php
$version = $this->versionRepository->findById($instance->versionId());
```
...it loads the **old version**'s snapshot, not the new one. The migration has no effect on execution.

**Two valid approaches:**

| Approach | Description | Complexity | Event Immutability |
|----------|-------------|------------|-------------------|
| A: `ProcessInstanceMigratedEvent` | New domain event applied to the aggregate. Engine replays it and overwrites `versionId`. Read model projection also handles it. | Medium | PRESERVED — new event added, old events untouched |
| B: Read-model-only | Accept that `version_id` in read model is cosmetic. Migration is documentation-only. Instances always run on their original version. | Low | PRESERVED — no event store change |
| C: Event store mutation | Mutate `ProcessStartedEvent` payload | Forbidden | VIOLATED — breaks Event Sourcing invariant |

**Recommendation:** Approach A (add `ProcessInstanceMigratedEvent`). This preserves event sourcing correctness and makes migration effective. Approach B is acceptable if the goal is only UI visibility of what version instances "should" be on.

**Confidence:** HIGH (code-verified)

### Gap 2: No Frontend Admin Migration UI

No Vue page, dialog, or component exists for admins to:
- View version history for a process definition (versions list)
- See how many running instances are on each version
- Trigger migration of running instances to a target version

The backend API is complete (`GET /versions`, `POST /versions/{id}/migrate`). The Pinia store has `versions` ref and `fetchVersions` action. What's missing:
- A version history panel/dialog in the Designer or ProcessDefinitions page
- A migrate button per version row
- A confirmation dialog showing migration impact (N instances will be migrated)
- Error display when migration is rejected (incompatible token positions)

**Where to add it:** Best integrated into `ProcessDesignerPage.vue` as a "Version History" panel or a dedicated `VersionHistoryDialog.vue` component, triggered from the Designer header.

**Confidence:** HIGH (frontend directories fully inspected)

### Gap 3: Instance Count per Version (Nice-to-Have for UI)

The `GET /versions` endpoint returns: `id`, `version_number`, `published_at`, `published_by`. It does NOT return the count of running instances on each version. The admin UI needs this to show "3 instances on v1, 0 on v2" before migrating.

**Fix:** Either:
- Extend `ProcessDefinitionVersionDTO` with an optional `running_instance_count` field
- Add a separate `GET /versions/running-instances` endpoint (overkill)
- Query `workflow_process_instances_view` inside `ListVersionsHandler` (join or subquery)

**Recommended approach:** Add a `runningInstanceCount` field populated in `ListVersionsHandler` via a DBAL subquery against `workflow_process_instances_view`.

**Confidence:** HIGH

### Gap 4: Compatibility Validation Scope

Current validation in `MigrateProcessInstancesHandler`:
```php
// Checks: active token node IDs must exist in target version snapshot
if (!in_array($token['node_id'], $targetNodeIds, true)) {
    throw new DomainException('Cannot migrate...');
}
```

This is **structurally correct** — it prevents migrating instances whose active tokens would be orphaned. What it does NOT check:
- Whether form fields required by the active node's transitions changed (new required fields with no default)
- Whether action keys (transition `action_key`) changed for active task nodes (user might be at a task with action "approve" that no longer exists in the new version)

**For Phase 06.1 scope:** The action key check is important for correctness. Form field validation is optional (the engine doesn't enforce required fields on completion, only on UI).

**Recommended additional check:**
```php
// For each active token at a task node, verify the outgoing transition action_keys still exist
$targetGraph = ProcessGraph::fromSnapshot($snapshot);
foreach (activeTokensAtTaskNodes) {
    foreach (currentVersionOutgoingTransitions[$token['node_id']]) {
        if (actionKey not in targetGraph->outgoingTransitions($token['node_id'])) {
            throw DomainException('Action key no longer available...');
        }
    }
}
```

**Confidence:** MEDIUM (based on WorkflowEngine analysis — `executeAction` requires matching action_key)

## Architecture Patterns

### Pattern 1: ProcessInstanceMigratedEvent (for Gap 1, Approach A)

**What:** New domain event that allows the aggregate to update its `versionId` field.

```php
// New event
final readonly class ProcessInstanceMigratedEvent extends DomainEvent
{
    public function __construct(
        public string $processInstanceId,
        public string $fromVersionId,
        public string $toVersionId,
        public string $migratedBy,
    ) {}
}
```

**In ProcessInstance:**
```php
public function migrateToVersion(ProcessDefinitionVersionId $newVersionId, string $migratedBy): void
{
    $this->recordThat(new ProcessInstanceMigratedEvent(
        processInstanceId: $this->id->value(),
        fromVersionId: $this->versionId->value(),
        toVersionId: $newVersionId->value(),
        migratedBy: $migratedBy,
    ));
}

// In apply():
$event instanceof ProcessInstanceMigratedEvent => $this->applyMigrated($event),

private function applyMigrated(ProcessInstanceMigratedEvent $event): void
{
    $this->versionId = ProcessDefinitionVersionId::fromString($event->toVersionId);
}
```

**In MigrateProcessInstancesHandler:** Load each running instance from event store, call `migrateToVersion()`, save back (which appends event to store and dispatches for read model projection).

**Warning:** This changes the handler from a single DBAL UPDATE to N event store appends (one per running instance). For definitions with many instances (say 1000), this is a slow operation. Consider making this async or batched.

**Confidence:** HIGH (follows existing Event Sourcing pattern in codebase)

### Pattern 3: Running Instance Count in ListVersionsHandler

```php
// In ListVersionsHandler, after loading versions:
$versionIds = array_map(fn($v) => $v->id()->value(), $versions);

// Single query for all counts
$countRows = $this->connection->fetchAllAssociative(
    'SELECT version_id, COUNT(*) as cnt
     FROM workflow_process_instances_view
     WHERE version_id = ANY(:ids) AND status = :status
     GROUP BY version_id',
    ['ids' => $versionIds, 'status' => 'running'],
    ['ids' => ArrayParameterType::STRING]
);
$countsByVersion = array_column($countRows, 'cnt', 'version_id');

// Inject into DTOs
return array_map(
    fn($v) => ProcessDefinitionVersionDTO::fromEntity($v, (int)($countsByVersion[$v->id()->value()] ?? 0)),
    $versions,
);
```

**Note:** `ListVersionsHandler` currently uses only the ORM repository, not DBAL. To add the count query, inject `Connection $connection` alongside the repository. This is the established project pattern (see `MigrateProcessInstancesHandler`, `GetProcessInstanceHandler`).

**Confidence:** HIGH

### Anti-Patterns to Avoid

- **Mutating event store records** — never UPDATE `workflow_process_events`. Events are immutable.
- **Rebuilding versioning from scratch** — `ProcessDefinitionVersion`, the snapshot mechanism, and the migration endpoint all exist. Only extend them.
- **Loading all process instances into PHP for migration** — use DBAL for bulk operations when possible. For event-sourced migration (Approach A), iterate instances but keep the loop efficient.
- **Putting running instance count in a separate API call** — add it to the existing versions DTO to avoid N+1 calls from the frontend.

## Common Pitfalls

### Pitfall 1: Read Model vs. Event Store Mismatch After Migration

**What goes wrong:** Migration updates `workflow_process_instances_view.version_id` but the engine reconstitutes instances from the event store with the original `versionId` from `ProcessStartedEvent`.

**Why it happens:** Event Sourcing + CQRS — the read model is a projection, not the source of truth for the write model.

**How to avoid:** Use `ProcessInstanceMigratedEvent` (Approach A) so the aggregate state is updated correctly. Or explicitly accept that migration is read-model-only (Approach B, cosmetic only).

**Warning signs:** After migration, tasks still complete against old version graph. This is detectable by checking if `CompleteTaskNodeHandler` loads the old version's snapshot after migration.

### Pitfall 2: N Event Store Appends Blocking HTTP Request

**What goes wrong:** If a definition has 500 running instances, migrating all of them synchronously in one HTTP request takes seconds and may time out.

**Why it happens:** Each `EventSourcedProcessInstanceRepository::save()` call does: N event store inserts + N event bus dispatches.

**How to avoid:** For Phase 06.1, accept synchronous for small datasets (< 100 instances typical in dev). Add a counter to the response (`migratedCount`). If the dataset grows, add async dispatch in a later phase.

### Pitfall 3: Doctrine DBAL `ArrayParameterType` for PostgreSQL `ANY(:ids)`

**What goes wrong:** Using standard `?` placeholder for array parameters with DBAL fails silently or throws.

**Why it happens:** DBAL requires `ArrayParameterType::STRING` as the type parameter for array binding.

**How to avoid:**
```php
use Doctrine\DBAL\ArrayParameterType;
$conn->fetchAllAssociative(
    'SELECT version_id, COUNT(*) FROM ... WHERE version_id = ANY(:ids)',
    ['ids' => $versionIdArray],
    ['ids' => ArrayParameterType::STRING],
);
```

**Confidence:** HIGH (pattern used in project's DBAL queries)

### Pitfall 4: `ProcessGraph::fromSnapshot()` Requires `{ nodes, transitions }` Shape

**What goes wrong:** Passing the raw `nodesSnapshot()` array directly where a flat node array is expected.

**Why it happens:** The snapshot is `{ nodes: [...], transitions: [...] }` — not a flat array. This was an actual bug found in Phase 6 (see 06-02-SUMMARY.md).

**How to avoid:** Always access `$snapshot['nodes']` and `$snapshot['transitions']` explicitly. `ProcessGraph::fromSnapshot()` handles the top-level keys correctly.

public function save(ProcessInstance $processInstance): void
{

    $currentVersion = $processInstance->version();

    foreach ($uncommittedEvents as $i => $event) {
        $this->eventStore->append($processInstance->id()->value(), $event, $currentVersion + $i + 1);
    }

    foreach ($uncommittedEvents as $event) {
        $this->eventBus->dispatch($event);
    }

    $processInstance->clearUncommittedEvents();
}

## Open Questions

1. **Migration semantics: Approach A or B?**
   - What we know: Current implementation is read-model-only (Approach B). Engine uses event-sourced versionId.
   - What's unclear: Whether the goal is actual version-aware execution after migration, or just administrative record-keeping.
   - Recommendation: **Clarify with user before planning.** If users expect migrated instances to actually run against the new version, Approach A is required. If migration is informational only, Approach B is sufficient and already done.

2. **Migration scope: all running instances vs. selective?**
   - What we know: Current API migrates ALL running instances of a definition to target version.
   - What's unclear: Whether admins should be able to select specific instances to migrate.
   - Recommendation: Start with all-or-nothing (current behavior). Selective migration can be added later.

3. **Where to surface the admin migration UI?**
   - Options: (a) Designer page sidebar, (b) ProcessDefinitionsPage row action, (c) dedicated admin page
   - Recommendation: Designer page — it's where version state is most contextually relevant.

4. **Should `EventSerializer` know about `ProcessInstanceMigratedEvent`?**
   - What we know: `EventSerializer` has explicit `match` arms for all events. Adding `ProcessInstanceMigratedEvent` requires adding serialize/deserialize cases.
   - Recommendation: Yes — follow the established pattern in `EventSerializer.php`.
