Enhance `useCanvasValidation` to accept a third parameter: `definition: Ref<ProcessDefinitionDetailDTO | null>` (optional, nullable). This allows the composable to check transition metadata from the API definition alongside the vue-flow graph structure.

Add a new validation check inside `validationErrors` computed:
1. If `definition` is provided, iterate over `definition.transitions`
2. For each transition that has `form_fields` array with length > 0 AND `action_key` is null or empty string:
   - Push a warning: `{ key: 'validationTransitionNoActionKey', params: { name: transition.name || sourceNodeLabel } }`
   - The source node label can be found from `nodes.value.find(n => n.id === transition.source_node_id)?.data.label`
3. Also check for DUPLICATE action_keys on transitions from the same source task node:
   - Group outgoing transitions per source task node
   - If two or more transitions from the same source have the same non-null action_key, push: `{ key: 'validationDuplicateActionKey', params: { name: sourceNodeLabel, actionKey: duplicatedKey } }`

Update the function signature:
```typescript
export function useCanvasValidation(
  nodes: Ref<Node[]>,
  edges: Ref<Edge[]>,
  definition?: Ref<ProcessDefinitionDetailDTO | null>,
```

  <action>
The research identified Pitfall 1: after NodePropertyPanel saves, the local node.data is updated but `props.definition` (from the store) is NOT re-fetched. This means `definition.transitions` and `definition.nodes` may become stale. The TransitionPropertyPanel reads `form_fields` from `definition.transitions`, so if the definition is stale, previously saved transition data may not reflect.

Fix in WorkflowDesigner.vue:

1. After `updateNodeData()` successfully saves to the API, emit a `definition-changed` event or call a definition refresh function. The simplest approach: add a `refetch` emit to tell the parent page to reload the definition from the store.

   Add to the emit definition:
   ```typescript
   const emit = defineEmits<{
     // ... existing emits
     'definition-changed': []
   }>()
   ```

   At the end of the successful `updateNodeData()` try block (after local `node.data` update), emit:
   ```typescript
   emit('definition-changed')
   ```

2. Similarly, in the TransitionPropertyPanel's `save()` flow: after the `emit('update', ...)` call returns, the WorkflowDesigner should also emit `definition-changed`. The TransitionPropertyPanel already emits `update` which WorkflowDesigner handles in `onEdgeUpdate`. In `onEdgeUpdate`, after updating the local edge data, also emit `definition-changed`:
   ```typescript
   function onEdgeUpdate(data: { name: string; condition_expression: string | null }) {
     // ... existing logic
     emit('definition-changed')
   }
   ```

3. In the parent page that hosts WorkflowDesigner (ProcessDefinitionDetailPage.vue or similar), listen for `@definition-changed` and re-fetch the definition from the store:
   ```typescript
   async function onDefinitionChanged() {
     await definitionStore.fetchDefinition(orgId, definitionId)
   }
   ```

   Find the parent page by searching for the WorkflowDesigner usage. Wire the event handler.

This ensures that after any node or transition save, the full definition (including all transitions with their form_fields and action_keys) is refreshed from the backend, keeping the local vue-flow state in sync with the API state.
