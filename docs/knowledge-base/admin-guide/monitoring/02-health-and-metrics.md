## Огляд

Procivo включає вбудовану систему моніторингу: health check ендпоінти для перевірки стану сервісів, Prometheus метрики для збору даних про продуктивність та Grafana дашборди для візуалізації. Також налаштовано автоматичне резервне копіювання PostgreSQL.

## Health Check ендпоінти

Health check ендпоінти доступні без авторизації (за межами JWT firewall) для використання балансувальниками навантаження та системами моніторингу.

| Ендпоінт | Перевіряє |
|----------|-----------|
| `/health` | Загальний стан системи |
| `/health/db` | Підключення до PostgreSQL |
| `/health/redis` | Підключення до Redis |
| `/health/rabbitmq` | Підключення до RabbitMQ |

### Формат відповіді

Кожен ендпоінт повертає JSON:

```json
{
  "status": "ok",
  "checks": {
    "database": "ok",
    "redis": "ok",
    "rabbitmq": "ok"
  }
}
```

- **Статус 200** -- все працює нормально
- **Статус 503** -- один або більше сервісів недоступні

## Prometheus метрики

Ендпоінт `/metrics` надає метрики у форматі Prometheus:

### Основні метрики

| Метрика | Опис |
|---------|------|
| `http_request_duration_seconds` | Час обробки HTTP запитів |
| `http_requests_total` | Загальна кількість HTTP запитів |
| `http_errors_total` | Кількість помилок (4xx, 5xx) |
| `messenger_queue_depth` | Глибина черги повідомлень |

### Збереження метрик

Prometheus використовує Redis як адаптер для збереження метрик між PHP запитами (PHP є stateless -- кожен запит створює новий процес).

## Grafana дашборди

Grafana доступна на порту **3001** (порт 3000 зайнятий Mercure).

### Доступ

1. Відкрийте `http://localhost:3001` у браузері
2. Увійдіть з обліковими даними за замовчуванням
3. Перейдіть до розділу "Dashboards"

### Наявні дашборди

- **Request Overview** -- загальна статистика запитів, час відповіді, коди помилок
- **Queue Monitoring** -- стан черг RabbitMQ, глибина та швидкість обробки
- **Database Performance** -- метрики PostgreSQL, час запитів

## Резервне копіювання PostgreSQL

Автоматичне резервне копіювання налаштовано через окремий Docker-контейнер:

### Розклад

| Тип | Частота | Збереження |
|-----|---------|-----------|
| Щоденний | Кожен день | S3 з префіксом `daily/` |
| Щотижневий | Кожну неділю | S3 з префіксом `weekly/` |
| Щомісячний | 1-го числа | S3 з префіксом `monthly/` |

### Механізм

- Контейнер на базі `postgres:18-alpine` (нативний `pg_dump`)
- Бекапи завантажуються в S3 (LocalStack у dev, AWS S3 у production)
- Ротація через S3 lifecycle policies
- Перший бекап виконується при запуску контейнера для перевірки

## Корисні поради

- Налаштуйте зовнішній моніторинг на `/health` для оперативних сповіщень
- Використовуйте Grafana алерти для порогових значень (час відповіді > 1s, черга > 100)
- Перевіряйте `/health/db` та `/health/redis` окремо при діагностиці проблем
- Регулярно перевіряйте, що бекапи створюються -- завантажте тестовий бекап та відновіть
